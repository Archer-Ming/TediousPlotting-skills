# 选图指南——选择合适的可视化习语

本指南帮助你根据两个核心问题选图：

1. **数据是什么结构？**（层级？网络？流向？多维属性？地理？）
2. **用户想用图做什么？**（比较占比？发现模式？下钻？追溯流向？）

当用户描述他们的数据与目标时，先走完本指南，再给出 1-2 个建议方案，**不要静默直接挑一个**。

---

## 快速决策树

```
主导数据结构是什么？

├── 层级结构（树、父子、嵌套类目）
│   ├── 展示部分-整体 + 支持下钻 → 旭日图 SUNBURST
│   └── 强调多叶节点占比、紧凑布局 → 矩形树图 TREEMAP
│
├── 网络/图谱（实体 + 连接，无固定层级）
│   ├── 关注连接结构、寻找聚类 → 力导向图 FORCE-DIRECTED
│   └── 固定类目之间的双向流动 → 和弦图 CHORD
│
├── 流向/过程（X 从哪里来到哪里去）
│   └── 桑基图 SANKEY
│
├── 多维记录（每条目有多个属性）
│   ├── ≤ 8 维、少量条目、对比 profile → 雷达图 RADAR
│   └── 4-15 维、多条目、寻找规律 → 平行坐标 PARALLEL COORDINATES
│
└── 地理（数据关联到行政区域/坐标）
    └── 地理分级填色图 CHOROPLETH
```

---

## 模板详解

每张图说明：
- **何时使用**（数据形态 + 分析任务）
- **强项**（感知优势）
- **弱项**（局限与扩展上限）
- **资源路径**（模板位置）

---

### 旭日图 Sunburst — `assets/sunburst/`

**何时使用：**
- 数据是 2-5 层的树
- 用户想看多尺度的部分-整体拆分
- 用户想下钻到某个子树

**强项：**
- 角度编码数值（最精确的定量通道之一）
- 一次性展现完整层级
- 点击下钻 + 中心点回退，既能聚焦又保留径向框架

**弱项：**
- 层级超过 5 层，外环就细到看不清
- 跨层级比较面积会误导（外环单位角度面积更大）
- 严重失衡的树（一个子树占 90% 体量）会变成单一扇形 + 一堆碎片

**扩展上限：** 3-5 层，每层 ≤ 20 个子节点，总叶节点 ≤ 200。

**示例数据结构：**
```javascript
{
  name: "Root",
  children: [
    { name: "A", children: [
      { name: "A1", value: 10 },
      { name: "A2", value: 15 }
    ]},
    { name: "B", value: 20 }
  ]
}
```

### 矩形树图 Treemap — `assets/treemap/`

**何时使用：**
- 数据是树状的，但你更关心**叶节点的大小**而非层级本身
- 空间效率很重要（在固定区域里放很多条目）
- 用户想一眼看出"什么最大"

**强项：**
- 面积编码数值，布局紧凑无浪费
- 位置承载类目信息（同级聚拢）
- "什么主导了整体"这类任务表现极佳

**弱项：**
- 层级本身在 2-3 层后就难读了（嵌套矩形看着眼花）
- 面积比较没有角度或长度精确
- 长宽比变化大，跨条目比较有噪声

**扩展上限：** 2-3 层，可同屏显示约 500 个叶节点。

**示例数据结构：** 与旭日图相同（树状）。

---

### 桑基图 Sankey — `assets/sankey/`

**何时使用：**
- 数据描述流：物质/能量/资金从源头经过若干阶段流到汇点
- 用户想看到流量在传递中守恒
- 2-5 个阶段，每阶段节点数量有限

**强项：**
- 缎带宽度精确编码量级
- 路径连接清晰展示分流与合流
- 回答"从哪里来"、"到哪里去"的任务极佳

**弱项：**
- 节点增多时交叉飞速增加，可读性迅速崩盘
- 没有箭头或颜色辅助时，方向不易分辨
- 不支持循环流（X → Y → X）

**扩展上限：** 2-5 阶段，每阶段 ≤ 15 节点，总连接 ≤ 100。

**示例数据结构：**
```javascript
{
  nodes: [
    { name: "Source A" },
    { name: "Process 1" },
    { name: "Sink X" }
  ],
  links: [
    { source: "Source A", target: "Process 1", value: 30 },
    { source: "Process 1", target: "Sink X", value: 30 }
  ]
}
```

### 力导向图 Force-Directed — `assets/force-directed/`

**何时使用：**
- 数据是通用图（节点 + 边），无内在层级
- 用户想发现社群、关键节点、桥梁
- 网络结构本身就是洞察对象

**强项：**
- 布局自动揭示社群结构
- 拖拽节点是强大的探索手段
- 能表达层级无法承载的关系（环、多父）

**弱项：**
- 布局非确定性——同样数据每次运行位置不同
- 超过 ~200 节点会变成"毛球"
- 节点密集时标签互相遮挡

**扩展上限：** ≤ 200 节点、≤ 500 边，理想 < 50 节点。

**示例数据结构：**
```javascript
{
  nodes: [{ id: "A", group: 1 }, { id: "B", group: 2 }, ...],
  links: [{ source: "A", target: "B", value: 3 }, ...]
}
```

### 和弦图 Chord — `assets/chord/`

**何时使用：**
- 数据是方阵：每个实体可与其他任意实体相连
- 连接是双向或对称的（贸易、迁徙、引用）
- 类目固定且不太多（≤ 20）

**强项：**
- 所有两两关系一次呈现
- 缎带宽度编码流量大小
- 径向布局节省空间

**弱项：**
- 超过 15 类目缎带就开始严重重叠
- 方向难以分辨（哪边是"出"？）
- 自循环（X → X）难以表达

**扩展上限：** 最佳 6-15 类目，超过 20 几乎崩塌。

**示例数据结构：**
```javascript
{
  names: ["A", "B", "C", "D"],
  matrix: [
    [0,   5,  3,  2],   // A → 其他
    [4,   0,  6,  1],   // B → 其他
    [2,   3,  0,  7],
    [1,   4,  5,  0]
  ]
}
```

### 雷达图 Radar — `assets/radar/`

**何时使用：**
- 在相同的 4-8 个属性上比较 2-5 个条目
- 每个属性有可比较的量纲（0-100、评分、星级）
- 用户想比较 profile 形状，而非绝对值

**强项：**
- 多边形形状成为每个条目的"指纹"
- 同一条目跨多个评分维度的快速比较
- 容易看出强项与弱项

**弱项：**
- 不能精确比较绝对值（面积取决于轴的顺序）
- 量纲不可比时会误导
- 超过 4 个条目重叠会变得混乱

**扩展上限：** 2-5 条目，4-8 个轴。

**示例数据结构：**
```javascript
{
  axes: ["速度", "动力", "舒适", "续航", "价格"],
  items: [
    { name: "条目 A", values: [80, 60, 90, 70, 40] },
    { name: "条目 B", values: [60, 90, 50, 80, 60] }
  ]
}
```

### 平行坐标 Parallel Coordinates — `assets/parallel-coords/`

**何时使用：**
- 多记录（50-1000 条），每条有多个数值属性（4-15 维）
- 用户想在高维空间里找聚类、相关性、离群点
- 用户希望对单个轴进行刷选（brush）过滤

**强项：**
- 单个画布上承载多维度
- 轴上拖拽刷选可交互式过滤数据集
- 轴间交叉揭示负相关；平行线揭示正相关

**弱项：**
- 容易拥挤——数百条重叠线让人难辨
- 轴的顺序强烈影响可见的规律（重排轴是必修课）
- 不能很好展示类别属性（需要单独编码）

**扩展上限：** 50-2000 记录，4-15 个轴。

**示例数据结构：**
```javascript
[
  { name: "条目 1", speed: 80, power: 60, comfort: 90, ... },
  { name: "条目 2", speed: 60, power: 90, comfort: 50, ... },
  ...
]
```

### 地理分级填色 Choropleth — `assets/choropleth/`

**何时使用：**
- 数据与地理区域绑定（国家、省、县）
- 每个区域一个数值
- 用户想看地理空间分布

**强项：**
- 即时的空间识别（人人认得自己国家的形状）
- 颜色梯度横扫地图，揭示地理聚集
- 习语熟悉，认知负担低

**弱项：**
- 地图上区域**面积**会与数据大小混淆（俄罗斯永远看着很重要）
- 不能精确读数值——颜色辨别能力约 5 级
- 地图投影选择会失真

**扩展上限：** 同屏最多约 200 区域。

**示例数据结构：**
```javascript
{
  geojson: { /* GeoJSON FeatureCollection */ },
  values: { "USA": 100, "CHN": 95, "DEU": 60, ... }   // 按区域 ID 键入
}
```

## 怎么向用户推荐

当用户描述他们的数据与目标时，给出 **1-2 个候选 + 明确理由**。格式参考：

> 根据你描述的——[复述数据 + 任务]——我建议：
>
> **首选：[模板 X]**——[理由：为什么贴合]。最适合 [用户陈述的任务]。
>
> **备选：[模板 Y]**——[同样问题的另一个解法]。如果 [条件] 则选这个。
>
> 你想要哪个？或者我帮你决定？

不要列全 8 个——那等于把决策推给用户。两个有理有据的候选给用户选择权而不至于过载。

## 反模式——什么时候**不要**用这些模板

- 用户只想要"一张月销售图" → 建议条形/折线图（不在本 skill 内，需说明）
- 用户有 2 个变量想看相关 → 散点图（不在本 skill 内）
- 用户有时间序列 → 折线图（不在本 skill 内）
- 用户要精确读出数值 → 条形图或表格更准确
- 用户要"信息图" → 本 skill 做的是可交互的网页可视化；静态印刷品需另想方式
